  <behavior>

    - Stream first replays persisted events where `sequence > cursor`, then subscribes to live fanout.
    - Cursor resolution accepts `afterSequence` query parameter and `Last-Event-ID` header; canonical cursor is numeric sequence.
    - SSE event `id` equals the event sequence and `event` name equals the event type wire name.
  </behavior>
  <action>
    Implement SSE support with Spring MVC `SseEmitter`.
    - `SseRunEventFanout` implements Infrastructure `RunEventFanout`; keep an in-memory concurrent map keyed by `runId` to active subscribers. On publish, send only to subscribers for that run; remove subscribers on send failure, completion, timeout, or error. This fanout is not the durable source of truth.
    - `RunEventStreamService.replayThenSubscribe(RequestContext context, String sessionId, String runId, long afterSequence, String lastEventId)` resolves the cursor: if `Last-Event-ID` is a positive long, use it; otherwise use `afterSequence`; then calls `RunQueryService.listEvents(..., cursor, 500)` and sends replay events in ascending sequence before registering for live events.
    - `RunEventStreamController` endpoint: `@GetMapping(path = "/api/sessions/{sessionId}/runs/{runId}/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)` with params `afterSequence` default `0` and header `Last-Event-ID` optional.
    - Build SSE events with `SseEmitter.event().id(Long.toString(event.sequence())).name(event.type()).data(event)` where `event` is `RunEventDto`.
